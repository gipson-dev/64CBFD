# Conker's Bad Fur Day Decompilation

A WIP decompilation of Conker's Bad Fur Day.

Note: To use this repository, you must already have a copy of the game.

Run `make -C conker progress` for an up-to-date local progress report.

# Folder structure

```text
conker/
├── conker/              # the "code" sub-project: init + game + debugger sections,
│   │                    #   cut out of the ROM and combined so they build like a
│   │                    #   normal single program (see conker/README.md)
│   ├── src/             # decompiled C source, one file per original object roughly
│   ├── include/         # libultra headers, struct/variable/function declarations
│   ├── asm/             # not-yet-matched function assembly (generated by splat,
│   │                    #   gitignored — recreated by `make -C conker extract`)
│   └── Makefile
├── config/              # yaml configs for extracting compressed asset chunks
│                        #   that don't have their own offsets table
├── tools/                # build tooling
│   ├── n64splat/         #   splits the ROM into segments            (submodule)
│   ├── asm-differ/       #   diffs your build against the real ROM   (submodule)
│   ├── asm-processor/    #   lets GLOBAL_ASM blocks live inside .c files (submodule)
│   ├── mips_to_c/        #   generates a first-pass C translation of MIPS asm (submodule)
│   ├── decomp-permuter/  #   randomized search for a byte-matching C variant (optional, gitignored)
│   └── *.py              #   custom scripts (compression, progress reporting, etc.)
├── ido/                  # recompiled IDO compiler — the actual toolchain used for
│                        #   byte-matching builds (Linux/macOS only, see below)
├── assets/               # extracted ROM assets (generated, gitignored)
├── *.yaml                # splat config, one per ROM version (us/eu/debug/ects)
├── Makefile              # top-level build orchestration
├── Dockerfile            # build environment definition
└── baserom.<version>.z64 # your own copy of the ROM — not included, see below
```

# Building

**You'll need:** your own copy of the **US** Conker's Bad Fur Day ROM, placed in the root of this repository and named `baserom.us.z64`. This repository does not, and will not, include the ROM itself.

**1. Get the toolchain.** The easiest path is [Docker](https://www.docker.com/products/docker-desktop) — it bundles everything the build needs. If you'd rather set things up natively, see the [Dockerfile](Dockerfile) for the exact package list; a plain **Ubuntu 20.04** (including via WSL on Windows) works well without Docker at all.

```sh
git clone --recursive <repository-url>
cd conker

# build the image once...
docker build . -t conker
# ...then use it whenever you want a shell with the toolchain available
docker run --rm -ti -v $(pwd):/conker conker bash
```

**2. Verify your ROM, then extract it.** `make check` confirms `baserom.us.z64` is byte-for-byte the ROM this repo expects (wrong dump = build will fail in confusing ways later, so don't skip this). `make extract` then splits the ROM into its component segments.

```sh
make check
make extract
```

**3. Build the code.** This is a two-part step because the ROM's code sections are cut out and compiled as their own sub-project (see [Folder structure](#folder-structure) above), then spliced back in:

```sh
make -C conker extract   # unpack the code sub-project's own asm/headers
make -C conker --jobs    # compile it — this is where byte-matching actually happens
make -C conker replace   # splice the freshly-compiled code back into the extracted ROM data
```

**4. Rebuild the ROM.**

```sh
make --jobs
```

If your code changes still match the original ROM byte-for-byte, you'll see:

```
build/conker.us.z64: OK
```

If it instead reports a mismatch, that's expected during active decomp work — it just means not everything is matched yet, not that something is broken. Use `tools/asm-differ` (see [Tools](#tools) below) to see exactly where your build diverges from the original.

# Development environment

If you're working from a checkout that lives on a Windows filesystem mounted into a Linux devcontainer/WSL2 (e.g. `/workspaces/<repo>` backed by a 9p/drvfs mount of a Windows drive), a few gotchas showed up in practice and are worth knowing about up front:

- **Line endings**: Windows git defaults to `core.autocrlf=true`, which converts LF to CRLF on checkout and makes the entire tree look modified under a Linux git. This repo now ships a `.gitattributes` forcing LF on all tracked text files; if you still see a wall of diffs on a fresh clone, also set `core.autocrlf=input` locally.
- **Symlinks**: if `core.symlinks` is `false` (common when git auto-detects a filesystem it thinks doesn't support them), tracked symlinks — e.g. `tools/splat_ext/rareunzip.py` — get checked out as plain text files containing the link target instead of real symlinks, which breaks splat's extension loading. Set `core.symlinks=true`; the 9p mount used by WSL2/devcontainers does support real symlinks once this is set.
- **Submodule "dubious ownership" errors**: caused by a uid mismatch between the container user and the mounted files. `.devcontainer/devcontainer.json`'s `postCreateCommand` now runs `git config --global --add safe.directory '*'` on every container (re)creation to fix this durably (a one-off fix in the running container doesn't survive a rebuild, since only the mounted workspace persists).
- **Pinned Python dependencies**: `tools/n64splat/requirements.txt` and `tools/mips_to_c`'s own `pyproject.toml` under-pin a couple of dependencies. A fresh `pip install` can pull a too-new `spimdisasm` (emits marker syntax `macro.inc`/`asm-processor` don't understand) or `pycparser` 3.x (removed a module `m2c` imports, breaking `mips_to_c --context`). Both are now pinned in the top-level `requirements.txt` (`spimdisasm==1.33.0`, `pycparser<3`) — if a similar tool breaks after a fresh install, suspect this same pattern first.

None of the above requires Docker/WSL specifically — a native Linux checkout is unaffected. Native Windows (no WSL/Docker at all) is not supported: the IDO compiler used for byte-matching (`ido/ido5.3_recomp/*`, from [decompals/ido-static-recomp](https://github.com/decompals/ido-static-recomp)) only targets Linux and macOS.

Progress is measured in bytes matched and functions matched. The detailed CSV files are regenerated with `make progress.csv` after a build.

There are three code sections within the ROM:

 - initialisation code + libultra; this is referred to as `init` and is translated to address `0x10000000`
 - core game code; this is referred to as `game` and is translated to address `0x15000000`
 - debugger code; referred to as `debugger` and translated to address `0x16000000`

In the `us`, `eu`, and `debug` ROMs, the `game` code is compressed. In the earlier `ects` ROM, it is not.

These sections are pulled out of the ROM and combined into the `conker/` code sub-project so they can be compiled consistently across versions.

## ROM layout

The layout of the ROM is still a work-in-progress. There are a number of sections within the ROM that are compressed with [gzip](https://tools.ietf.org/html/rfc1952) but have the standard header/trailer stripped and, instead, replaced with a 4-byte header containing the uncompressed data length. These sections are dubbed `rzip`.

Overview of US ROM shown below:
```
[header]  0000 0000 > 0000 0040 ; suggests libultra 2.0G
[ boot ]  0000 0040 > 0000 1000 ;
[ code ]  0000 1000 > 0004 2C50 ; init + libultra .text
[ ???? ]  0002 90D0 > ???? ???? ;
[ data ]  0002 C750 > 0002 C7A0 ; init + libultra .data
[ ???? ]  0002 C7A0 > 0004 2C50 ; μcode
[ rzip ]  0004 2C50 > 0018 6B50 ; game .text (compressed)
[ rzip ]  0018 8328 > 0019 C7D8 ; game .data (compressed)
[ code ]  0019 EA88 > 001A 2190 ; debugger .text
[ data ]  001A 2190 > 001A 37E0 ; debugger .data
[ rzip ]  001A 37E0 > 00AB 1950 ; compressed section (textures?)
[ offs ]  00AB 1950 > 00AB 1A40 ; table of asset offsets
[ rzip ]  00AB 1A40 > 03F8 B800 ; assets 00 thru assets 1C
[ ffff ]  03F8 B800 > 0400 0000 ; 0xff padding
```

### Compressed section(s)

There are a number of compressed sections within the ROM. The decompression/compression method is understood and generates matching results.

## Code sub-project build flow

The `conker/` directory is the code sub-project. From there, the usual loop is:

```sh
make extract
make --jobs
make replace
make -C ..
```

That extracts the code sections, compiles them, replaces the corresponding sections in the split ROM output, and rebuilds the full ROM from the repository root.

# Tools

## Custom tools

 - `rarezip/rareunzip`; python script to compress/decompress the compression format used in the ROM.

NOTE: `gzip` is used for compression rather than `zlib`; use the binary in `tools/` in order to get matching compression.

## Existing tools

This repo makes use of the following open-source tools without which, there would be no decomp:

 - [asm-differ](https://github.com/simonlindholm/asm-differ); compare assembly against the original ROM
 - [asm-processor](https://github.com/simonlindholm/asm-processor); allow `GLOBAL_ASM` wrappers to include assembly within the c files
 - [n64splat](https://github.com/ethteck/n64splat); split up the rom & much more...
 - [ido-static-recomp](https://github.com/Emill/ido-static-recomp); IDO compiler
 - [gzip](https://github.com/mkst/gzip); gzip; specifically with the pre-1.5 `memzero` behaviour
